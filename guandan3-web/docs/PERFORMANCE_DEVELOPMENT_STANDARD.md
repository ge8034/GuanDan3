# 性能开发标准

## 概述

本文档定义了掼蛋3.0项目的性能开发标准，包括性能测试、监控、优化和验收标准。目标是确保系统在20人并发场景下满足核心性能指标，并建立可持续的性能优化流程。

## 核心性能指标

### 必须满足的指标
1. **首屏加载时间**：≤ 2秒（3G网络模拟）
2. **游戏操作延迟**：≤ 100ms（P99百分位）
3. **API响应时间**：≤ 100ms（P95百分位）
4. **数据库查询时间**：≤ 50ms（平均响应时间）
5. **CPU使用率**：≤ 30%（20人并发，1 vCPU）
6. **内存使用**：≤ 512MB（峰值使用）

### 推荐指标
1. **Web Vitals**：LCP ≤ 2.5s，FID ≤ 100ms，CLS ≤ 0.1
2. **实时同步延迟**：≤ 60ms（P99百分位）
3. **错误率**：≤ 0.1%（HTTP错误率）
4. **吞吐量**：≥ 100请求/秒（20人并发）

## 性能测试标准

### 1. 单元性能测试
- **位置**：`src/test/performance/`
- **要求**：
  - 所有核心算法必须有性能测试
  - 测试执行时间阈值 ≤ 100ms
  - 内存使用必须在合理范围内
  - 必须包含边界情况和极端场景测试
- **工具**：Vitest + Performance API

### 2. 集成性能测试
- **位置**：`tests/e2e/perf-*.spec.ts`
- **要求**：
  - 关键路径必须包含Web Vitals指标验证
  - 必须测试端到端性能指标
  - 必须包含网络条件模拟（3G/4G）
  - 必须验证缓存策略有效性
- **工具**：Playwright + PerformanceObserver

### 3. 负载测试
- **位置**：`k6/load-test.js`
- **要求**：
  - 必须模拟真实用户行为模式
  - 必须支持20人并发场景
  - 必须测量吞吐量、响应时间、错误率
  - 必须包含阶梯式负载增加
- **工具**：k6 + 自定义指标

### 4. 压力测试
- **位置**：`scripts/load-test-node.js`
- **要求**：
  - 必须发现系统极限和瓶颈
  - 必须提供容量规划数据
  - 必须测试系统恢复能力
  - 必须验证降级策略
- **工具**：Node.js + 并发控制

## 性能监控标准

### 1. 客户端监控
- **必须监控的指标**：
  - Web Vitals（FCP, LCP, CLS, FID, TTFB, TTI）
  - 资源加载时间（JavaScript, CSS, 图片, 字体）
  - 用户交互延迟（点击、滚动、输入）
  - 内存使用情况
- **实现要求**：
  - 使用`src/lib/performance/performance-monitor.ts`
  - 关键页面必须启动性能监控
  - 性能数据必须上报到监控系统
  - 必须支持实时性能分析

### 2. 服务器端监控
- **必须监控的指标**：
  - API响应时间（P50, P95, P99）
  - 请求吞吐量（请求/秒）
  - 错误率（HTTP错误、业务错误）
  - 资源使用率（CPU、内存、网络）
- **实现要求**：
  - 所有API端点必须有性能监控
  - 必须记录慢查询日志
  - 必须实现请求追踪
  - 必须支持性能指标聚合

### 3. 数据库监控
- **必须监控的指标**：
  - 查询执行时间
  - 连接池使用率
  - 锁等待时间
  - 索引使用效率
- **实现要求**：
  - 必须启用Supabase监控
  - 必须配置慢查询告警
  - 必须定期分析查询性能
  - 必须优化低效查询

### 4. 实时监控
- **必须监控的指标**：
  - Supabase Realtime延迟
  - WebSocket连接稳定性
  - 消息传输成功率
  - 断线重连率
- **实现要求**：
  - 必须测量端到端实时延迟
  - 必须监控连接状态
  - 必须实现重连策略
  - 必须记录实时事件日志

### 2. 后端性能预算
- **API响应时间**：P95 < 100ms, P99 < 200ms
- **数据库查询**：单查询 < 50ms
- **内存使用**：峰值 < 512MB
- **网络带宽**：平均 < 1Mbps/用户

### 3. 实时性能预算
- **消息延迟**：P99 ≤ 60ms
- **连接建立时间**：≤ 500ms
- **消息成功率**：≥ 99.9%
- **重连时间**：≤ 3秒

## 性能优化标准

### 1. 代码级优化
- **算法优化**：
  - 必须选择时间复杂度最优的算法
  - 必须避免不必要的循环嵌套
  - 必须使用适当的数据结构
  - 必须实现缓存和记忆化
- **内存优化**：
  - 必须避免内存泄漏
  - 必须及时释放不再使用的对象
  - 必须使用对象池管理频繁创建的对象
  - 必须优化大对象的使用

### 2. 数据库优化
- **索引优化**：
  - 必须为频繁查询的字段创建索引
  - 必须使用复合索引优化多条件查询
  - 必须使用部分索引优化特定查询
  - 必须定期分析和优化索引
- **查询优化**：
  - 必须避免N+1查询问题
  - 必须使用JOIN代替多次查询
  - 必须优化子查询性能
  - 必须使用预编译语句

### 3. 网络优化
- **资源优化**：
  - 必须使用CDN加速静态资源
  - 必须启用HTTP/2多路复用
  - 必须使用Gzip/Brotli压缩
  - 必须实现资源懒加载
- **连接优化**：
  - 必须启用DNS预解析
  - 必须使用连接复用
  - 必须优化TCP连接参数
  - 必须实现连接保活

### 4. 缓存策略
- **客户端缓存**：
  - 必须使用Service Worker缓存静态资源
  - 必须实现合理的缓存失效策略
  - 必须支持离线访问
  - 必须处理缓存更新
- **服务器端缓存**：
  - 必须使用Redis缓存热点数据
  - 必须实现分布式缓存
  - 必须处理缓存一致性
  - 必须监控缓存命中率

## 性能测试执行标准

### 1. 本地开发阶段
```bash
# 必须运行的性能测试
npm run test:performance      # 运行单元性能测试
npm run test:e2e:perf        # 运行E2E性能测试
npm run perf:budget          # 检查性能预算
```

### 2. CI/CD流水线
```yaml
# .github/workflows/performance.yml
# 必须包含的性能测试阶段
- name: Run unit performance tests
  run: npm run test:performance

- name: Run load test
  run: npm run test:load

- name: Generate performance report
  run: npm run perf:report
```

### 3. 生产环境监控
- **必须实现的监控**：
  - 7×24小时性能监控
  - 实时告警机制
  - 性能趋势分析
  - 容量规划预测
- **必须配置的告警**：
  - API响应时间超过阈值
  - 错误率超过阈值
  - 资源使用率超过阈值
  - 实时延迟超过阈值

## 性能验收标准

### 1. 测试验收标准
- **单元性能测试**：通过率必须达到100%
- **集成性能测试**：通过率必须达到95%以上
- **负载测试**：必须满足20人并发性能指标
- **压力测试**：必须发现系统极限并提供优化建议

### 2. 监控验收标准
- **监控覆盖率**：必须达到100%
- **告警准确率**：必须达到95%以上
- **问题发现时间**：必须小于5分钟
- **问题解决时间**：必须小于30分钟

### 3. 优化验收标准
- **性能改进**：必须量化性能改进效果

## 性能问题处理流程

### 1. 问题发现
1. **监控告警**：监控系统发现性能问题
2. **用户反馈**：用户报告性能问题
3. **测试发现**：性能测试发现性能问题
4. **代码审查**：代码审查发现潜在性能问题

### 2. 问题分析
1. **数据收集**：收集相关性能数据
2. **根因分析**：分析性能问题根本原因
3. **影响评估**：评估性能问题影响范围
4. **优先级确定**：确定问题修复优先级

### 3. 问题解决
1. **方案设计**：设计性能优化方案
2. **方案评审**：评审性能优化方案
3. **方案实施**：实施性能优化方案
4. **效果验证**：验证性能优化效果

### 4. 问题预防
1. **经验总结**：总结性能问题处理经验
2. **流程优化**：优化性能问题处理流程
3. **监控增强**：增强相关性能监控
4. **团队培训**：培训团队避免类似问题
