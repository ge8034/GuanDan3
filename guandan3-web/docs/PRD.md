## 1. 产品概述

### 1.1 产品简介

掼蛋3（GuanDan3）是一款基于Web技术的在线掼蛋游戏平台，采用前后端分离架构，支持多人实时对战和单人练习模式。游戏使用两副扑克牌（108张），由4名玩家分成两队进行对抗，结合了"跑得快"和"升级"的游戏特点。

### 1.2 产品定位

- **目标用户**: 扑克牌游戏爱好者，年龄18-45岁
- **核心价值**: 随时随地、打开即玩的掼蛋游戏体验
- **差异化优势**: 
  - 无需注册，匿名登录即可开始游戏
  - 轻量级Web应用，支持桌面和移动设备
  - 实时对战，低延迟响应
  - AI练习模式，支持单机游戏

### 1.3 核心价值主张

| 价值维度 | 描述 | 实现方式 |
|---------|------|----------|
| 便捷性 | 进入即玩，无需注册 | Supabase匿名认证 |
| 实时性 | 低延迟多人对战 | Supabase Realtime |
| 可访问性 | 跨平台Web应用 | Next.js响应式设计 |
| 稳定性 | 断线重连，状态恢复 | 心跳检测+状态同步 |
| 可扩展性 | 模块化架构，易于扩展 | 组件化+状态管理 |

### 3.2 用户认证

#### 3.2.1 匿名登录

**功能描述**: 用户首次访问时自动创建匿名账户，无需注册表单

**交互流程**:
```
访问网站 → 自动创建匿名会话 → 生成默认昵称 → 进入大厅
```

**技术实现**:
- 使用 Supabase Anonymous Sign-in API
- 生成唯一 UID 用于 RLS 隔离
- 默认昵称格式：`玩家{随机数字}`

**验收标准**:
- [x] 首次访问自动登录
- [x] 跨会话保持登录状态
- [x] RLS 策略正确隔离用户数据

#### 3.2.2 个人资料管理

**功能描述**: 用户可修改昵称和头像

**字段**:
- 昵称（2-20字符）
- 头像URL（可选）
- 创建时间（只读）

### 3.3 游戏大厅

#### 3.3.1 房间列表

**功能描述**: 展示可用房间，支持刷新和筛选

**显示字段**:
- 房间名称
- 游戏模式（PVP4 / 练习）
- 玩家数量（x/4）
- 房间状态（空闲/游戏中）

**交互操作**:
- 刷新列表
- 筛选模式
- 快速加入（自动选择有空位的房间）

#### 3.3.2 创建房间

**功能描述**: 创建新游戏房间

**配置项**:
- 房间名称（可选，默认格式："房间{随机码}"）
- 游戏模式：PVP4（4人）/ 练习（1v3 AI）
- 可见性：公开 / 私密

**权限**: 房主拥有特殊权限
- 开始游戏
- 踢出玩家
- 设置房间密码（私密房间）

#### 3.3.3 加入房间

**功能描述**: 加入已有房间

**流程**:
1. 选择房间
2. 选择座位号（0-3）
3. 等待其他玩家
4. 准备/取消准备
5. 等待房主开始游戏

**座位分配规则**:
- 坐位0和2为队友（A队）
- 坐位1和3为队友（B队）
- 座位号为逆时针顺序

### 3.4 游戏对局

#### 3.4.1 发牌流程

**规则**:
- 使用两副扑克牌，共108张
- 每人发27张牌
- 留6张底牌
- 确定级牌（从2开始）

**状态**:
```
waiting → dealing → playing → finished
```

#### 3.4.2 叫牌流程

**规则**:
- 从座位0开始，逆时针叫牌
- 叫牌者获得底牌，成为庄家
- 庄家的对家为守庄方
- 级牌作为当前局的王牌

#### 3.4.3 出牌流程

**规则**:
- 庄家先出牌
- 按逆时针顺序轮流出牌
- 后家必须出相同牌型且更大的牌
- 无法出牌时选择"过牌"
- 一轮中其他三人都过牌后，出牌者获得新一轮出牌权

**牌型验证**:
- 单张
- 对子（两张相同）
- 三张（三张相同）
- 三带二（三张相同+两张任意）
- 顺子（五张或更多连续单牌，不含2和王）
- 连对（三对或更多连续对子）
- 钢板（两个连续三张）
- 炸弹（四张相同）
- 王炸（四张王：两张大王+两张小王）
- 逢人配（级牌可当作任意牌）

#### 3.4.4 结算流程

**获胜条件**:
- 一方两名玩家先出完牌，该方获胜

**结算规则**:
- 获胜方等级+1
- 双方先出完可连升2级
- 使用炸弹压制可跳级
- 根据出牌情况计算积分

### 3.5 练习模式

#### 3.5.1 AI设置

**配置**:
- 3个AI对手（座位1-3）
- AI难度：简单 / 中等 / 困难
- 自动开局，无需等待

**AI行为**:
- 基于规则的决策系统
- 支持基本牌型识别
- 简单的策略评估

### 3.6 聊天系统

#### 3.6.1 聊天功能

**消息类型**:
- 文字消息

- 表情包

**聊天范围**:
- 房间内广播
- 只能看到同一房间的消息

### 3.7 排行榜

#### 3.7.1 排名规则

**排名依据**:
- 总积分

- 游戏场次

**排行榜类型**:

## 6. 数据模型

### 6.1 核心实体

| 实体 | 表名 | 主键 | 关键字段 |
|------|------|------|----------|
| 用户 | users | id (UUID) | id, email, created_at |
| 资料 | profiles | uid (UUID) | uid, nickname, avatar_url |
| 房间 | rooms | id (UUID) | id, owner_uid, mode, status, visibility |
| 房间成员 | room_members | id (UUID) | room_id, member_type, seat_no, ready |
| 牌局 | games | id (UUID) | id, room_id, seed, status, current_seat |
| 手牌 | game_hands | (game_id, uid) | game_id, uid, hand (JSONB) |
| 出牌记录 | turns | id (UUID) | id, game_id, turn_no, seat_no, payload |
| 积分 | scores | id (UUID) | id, game_id, uid, delta, total_after |

### 6.2 实体关系图

```
users (1) ──<> (1) profiles
  │
  ├─(1)─<>─(N) rooms (作为 owner)
  │         │
  │         └─(1)─<>─(N) room_members
  │                   │
  │                   ├─(1)─<>─(N) games
  │                   │         │
  │                   │         ├─(1)─<>─(N) game_hands
  │                   │         ├─(1)─<>─(N) turns
  │                   │         └─(1)─<>─(N) scores
  │
  └─(1)─<>─(N) room_members (作为成员)
```

### 6.3 关键字段说明

#### rooms
- **mode**: `pvp4`（4人对战） / `pve1v3`（练习模式）
- **status**: `open`（空闲） / `playing`（游戏中） / `closed`（已关闭）
- **visibility**: `public`（公开） / `private`（私密）

#### room_members
- **member_type**: `human`（真人） / `ai`（AI）
- **seat_no**: 0-3（座位号）
- **ready**: 是否已准备

#### games
- **seed**: 随机种子，用于发牌
- **status**: `deal`（发牌中） / `playing`（游戏中） / `finished`（已结束）
- **state_public**: 公共游戏状态（所有玩家可见）
- **state_private**: 私有游戏状态（仅服务器可见）

## 9. 安全要求

### 9.1 认证授权

- **匿名认证**: 使用Supabase Anonymous Sign-in
- **RLS策略**: 所有表配置行级安全策略
- **JWT验证**: 每次请求验证JWT令牌

### 9.2 数据保护

- **输入验证**: 所有用户输入在边界验证
- **SQL注入防护**: 使用参数化查询
- **XSS防护**: React自动转义，CSP策略
- **CSRF防护**: SameSite Cookie

### 9.3 数据隔离

- **用户数据**: 只能访问自己的数据
- **房间数据**: 只能访问自己房间的数据
- **游戏数据**: 根据角色区分可见数据

|------|------|----------|
