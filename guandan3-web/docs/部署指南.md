## 🏗️ 部署架构概述

### 架构组件
- **前端应用**: Next.js 16 + React 19
- **进程管理**: PM2 集群模式
- **反向代理**: Nginx (HTTPS + HTTP/2)
- **CI/CD**: GitHub Actions
- **云平台**: Fly.io / 自建服务器
- **数据库**: Supabase (PostgreSQL)

### 部署环境
1. **开发环境**: 本地开发
2. **灰度环境**: canary.guandan3.com (10%-50%流量)
3. **生产环境**: guandan3.com (100%流量)

### 流量分配
```
用户请求 → Nginx负载均衡
           ├─ 90% → 生产环境 (PM2集群)
           └─ 10% → 灰度环境 (PM2集群)
```

## 🔧 环境准备

### 服务器要求
- **操作系统**: Ubuntu 20.04+ / CentOS 7+
- **Node.js**: v18.x
- **内存**: 最低2GB，推荐4GB+
- **存储**: 最低20GB SSD
- **网络**: 公网IP，开放80/443端口

### 必要软件安装
```bash
# Node.js 18
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt-get install -y nodejs

# PM2
sudo npm install -g pm2

# Nginx
sudo apt-get install -y nginx

# Git
sudo apt-get install -y git

# 其他工具
sudo apt-get install -y curl wget bc jq
```

### GitHub Secrets配置
在GitHub仓库设置中配置以下Secrets：

| Secret名称 | 说明 | 示例 |
|-----------|------|------|
| `SSH_PRIVATE_KEY` | SSH私钥，用于服务器连接 | `-----BEGIN RSA PRIVATE KEY-----...` |
| `REMOTE_HOST` | 服务器IP地址或域名 | `123.45.67.89` |
| `REMOTE_USER` | 服务器用户名 | `deploy` |
| `SLACK_WEBHOOK_URL` | Slack通知Webhook URL | `https://hooks.slack.com/...` |

### 环境变量配置
在服务器上创建 `.env.local` 文件：
```bash
# Supabase配置
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

# 应用配置
NODE_ENV=production
PORT=3000
```

## 🚀 灰度发布流程

### 方式一：使用GitHub Actions（推荐）

#### 1. 触发灰度发布
在GitHub仓库页面：
1. 进入 `Actions` 标签页
2. 选择 `Canary Deployment` workflow
3. 点击 `Run workflow`
4. 选择初始灰度流量（5%/10%/20%/50%）
5. 点击 `Run workflow` 按钮

#### 2. 自动化流程
GitHub Actions将自动执行：
- ✅ 代码检查（Lint + Type Check）
- ✅ 单元测试
- ✅ 安全审计
- ✅ 应用构建
- ✅ 部署到灰度环境
- ✅ 健康检查
- ✅ 性能监控（5分钟）
- ✅ 流量逐步增加
- ✅ 创建Canary Release

#### 3. 监控灰度环境
访问灰度环境：https://canary.guandan3.com

监控指标：
- 健康状态：`https://canary.guandan3.com/health`
- 错误率：应 < 5%
- 响应时间：应 < 3秒
- 用户反馈：收集用户使用体验

### 方式二：手动执行脚本

```bash
# 赋予脚本执行权限
chmod +x scripts/canary-deploy.sh

# 执行灰度发布（默认10%流量）
./scripts/canary-deploy.sh

# 指定初始流量
./scripts/canary-deploy.sh 20  # 20%流量
```

### 灰度发布配置参数
编辑 `scripts/canary-deploy.sh` 中的配置：
```bash
CANARY_PERCENT=10        # 初始流量百分比
MAX_PERCENT=100          # 最大流量百分比
INCREMENT=10            # 每次增加的流量
MONITOR_DURATION=300     # 每个流量级别的监控时长（秒）
ERROR_THRESHOLD=5        # 错误率阈值（%）
HEALTH_CHECK_URL="https://canary.guandan3.com/health"
```

## 🎯 全量发布流程

### 前置条件
- ✅ 灰度发布成功运行至少24小时

- ✅ 用户反馈良好
- ✅ 所有监控指标正常

### 方式一：使用GitHub Actions（推荐）

#### 1. 触发全量发布

2. 选择 `Full Production Deployment` workflow
3. 点击 `Run workflow`
4. 选择是否跳过灰度（通常不跳过）

6. 点击 `Run workflow` 按钮

- ✅ 预部署检查（Lint + Type Check + Tests）
- ✅ E2E测试

- ✅ Lighthouse CI性能测试
- ✅ 创建备份
- ✅ 部署到生产环境
- ✅ 更新Nginx配置（100%流量）
- ✅ 重启PM2进程
- ✅ 健康检查
- ✅ 性能验证
- ✅ 关键功能验证
- ✅ 发布后验证测试
- ✅ 创建Production Release
- ✅ Slack通知

### 方式二：手动执行脚本

```bash
# 赋予脚本执行权限
chmod +x scripts/full-deploy.sh

# 执行全量发布
./scripts/full-deploy.sh
```

## ✅ 发布后验证

### 自动化验证
全量发布后自动执行 `scripts/post-deploy-verification.sh`：

#### 验证项目
1. **健康检查** - 应用健康状态
2. **页面可访问性** - 所有关键页面
3. **静态资源加载** - CSS/JS/图片等
4. **API端点** - 所有API接口
5. **性能测试** - 加载时间和TTFB
6. **安全性测试** - HTTPS和安全头
7. **数据库连接** - 数据库连接状态
8. **实时连接** - WebSocket连接
9. **错误处理** - 404和错误页面
10. **并发测试** - 并发请求处理

#### 验证标准
- ✅ 所有健康检查通过
- ✅ 页面加载时间 < 3秒
- ✅ TTFB < 1秒
- ✅ 错误率 < 1%
- ✅ 安全头配置正确
- ✅ 数据库连接正常

### 手动验证清单
```bash
# 1. 健康检查
curl https://guandan3.com/health

# 2. 页面访问
curl -I https://guandan3.com/
curl -I https://guandan3.com/lobby
curl -I https://guandan3.com/room

# 3. API测试
curl https://guandan3.com/api/health

# 4. 性能测试
curl -o /dev/null -s -w "加载时间: %{time_total}s\nTTFB: %{time_starttransfer}s\n" https://guandan3.com/

# 5. SSL证书检查
curl -Iv https://guandan3.com 2>&1 | grep "SSL certificate"

# 6. PM2状态检查
pm2 status
pm2 logs guandan3-web --lines 50

# 7. Nginx状态检查
sudo systemctl status nginx
sudo nginx -t
```

## 🔄 回滚操作

### 自动回滚
发布失败时，GitHub Actions会自动执行回滚：
1. 恢复上一个稳定版本的备份
2. 重启PM2进程
3. 恢复Nginx配置
4. 执行健康检查
5. 发送Slack通知

### 手动回滚
```bash
# 方式一：使用PM2回滚
cd /var/www/guandan3-web
pm2 reload ecosystem.config.js --env production --update-env

# 方式二：从备份恢复
BACKUP_DIR=$(cat .last_backup)
cp -r "$BACKUP_DIR/.next" .
cp -r "$BACKUP_DIR/public" .
cp "$BACKUP_DIR/package.json" .
cp "$BACKUP_DIR/ecosystem.config.js" .
pm2 reload ecosystem.config.js --env production

# 方式三：Git回滚
git checkout <last-stable-commit-hash>
npm run build
pm2 reload ecosystem.config.js --env production
```

### 回滚验证
```bash
# 健康检查
curl https://guandan3.com/health

# 检查PM2日志
pm2 logs guandan3-web --lines 100

# 检查Nginx日志
sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log
```

## 📊 监控和告警

### 应用监控
```bash
# PM2监控
pm2 monit

# PM2 Web监控
pm2 web

# 查看进程状态
pm2 status
pm2 info guandan3-web
```

### 性能监控
```bash
# CPU和内存使用
pm2 show guandan3-web

# 系统资源监控
top
htop

# 磁盘使用
df -h
du -sh /var/www/guandan3-web
```

### 日志监控
```bash
# PM2日志
pm2 logs guandan3-web --lines 100
pm2 logs guandan3-web --err

# Nginx访问日志
sudo tail -f /var/log/nginx/access.log

# Nginx错误日志

tail -f /var/www/guandan3-web/logs/app.log
```

## 🔍 故障排查

### 常见问题

#### 1. 应用无法启动
```bash
# 检查PM2状态
pm2 status

# 查看错误日志
pm2 logs guandan3-web --err

# 检查端口占用
sudo netstat -tulpn | grep :3000

# 重启PM2
pm2 restart guandan3-web
```

#### 2. Nginx配置错误
```bash
# 测试Nginx配置
sudo nginx -t

# 查看Nginx错误日志
sudo tail -f /var/log/nginx/error.log

# 重载Nginx
sudo systemctl reload nginx
```

#### 3. 数据库连接失败
```bash
# 检查环境变量
cat .env.local | grep SUPABASE

# 测试数据库连接
curl https://your-project.supabase.co/health

# 检查Supabase状态
# 访问 https://supabase.com/status
```

#### 4. 性能问题
```bash
# 检查响应时间
curl -o /dev/null -s -w "%{time_total}" https://guandan3.com/

# 检查PM2进程资源使用
pm2 show guandan3-web

# 检查系统负载
uptime
top
```

#### 5. 内存泄漏
```bash
# 监控内存使用
watch -n 5 'pm2 show guandan3-web | grep memory'

# 重启应用释放内存
pm2 restart guandan3-web

# 启用PM2内存限制
# 编辑ecosystem.config.js
max_memory_restart: '1G'
```

### 紧急故障处理
```bash
# 1. 立即回滚到上一个稳定版本
./scripts/rollback-emergency.sh

# 2. 检查系统状态
systemctl status nginx
pm2 status
df -h
free -m

# 3. 收集诊断信息
pm2 logs guandan3-web --lines 500 > emergency-logs.txt
sudo journalctl -u nginx -n 500 > nginx-logs.txt

# 4. 联系运维团队
# 发送紧急通知到Slack
```
